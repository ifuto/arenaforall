00 / OVERVIEW

ローダーを、
ゲームの外へ。

Rsift は「Mod を読み込む道具」ではなく、Minecraft の実行境界そのものを設計し直すためのネイティブホストです。描画・計算・録画といった重い処理を、必要に応じて JVM の外へ逃がせます。

WHY RSIFT

Java の中で戦わない

従来のローダーは JVM 内でクラス変換やイベント配線を行います。Rsift は OS ネイティブプロセスとして先に立ち上がり、JVM を管理対象として扱います。

FOCUS

RsGraphics が本丸

公式プラグイン RsGraphics は、エンティティ密集場面や地形メッシュ、領域セーブなど、体感 FPS に直結する経路をネイティブ側から再設計します。目標は Sodium を超えること。

ECOSYSTEM

必要な能力だけを載せる

RsCalc（物理/AI）、RsReplay（キャプチャ）など、機能を native plugin として分離。使わないものを読み込まない構成を前提にしています。

01 / RSIFT CORE

Java の内側ではなく、
その前へ。

Rsift は描画 Mod ではありません。Minecraft が起動するより前に立ち上がり、JVM・メモリ・ネイティブプラグインの関係を再設計するための、純 Rust 製ローダーコアです。クラスパッチ、プロセス制御、オフヒープ IPC を一つのマスタープロセスに集約します。

01 / CONTROL

主従関係をネイティブ側へ。

rsift.exe が先に起動し、JVM を管理対象として生成します。実行優先度やコア割り当ての方針を、Java ヒープの外から設計できる構成です。

01

主従関係の完全逆転

従来のように JVM 内でローダーを実行するのではなく、純 Rust 製のネイティブ実行ファイルがマスターとして先行します。ゲームは「ホストされる側」です。

02

ヒープ非依存のパッチ

JVMTI フックを通じ、クラスが定義されるネイティブ層で処理。重い中間 Java オブジェクトの生成を避ける設計です。

03

オフヒープ・ゼロコピー IPC

Direct buffer とロックフリーリングバッファで、データ本体ではなくアドレスとサイズを共有する架け橋を作ります。

02 / OFFICIAL PLUGIN

← 概要に戻る

RsGraphics.

従来の描画最適化が抱える構造的な限界を越えるため、描画パイプラインをネイティブ層から再設計する Rsift 公式プラグインです。目標は明確で、Sodium を超えること。特に「見えていない動的オブジェクト」まで処理し続ける密集場面を主戦場にしています。

01 / GPU-DRIVEN

CPU の準備待ちを、描画の中心にしない。

広大なワールドの形状と位置情報を GPU メモリに保持し、可視・不可視の判断を GPU 側へ寄せる構造を目指します。CPU が細かい命令を準備し続ける旧来のボトルネックを減らします。

02 / MULTI-LAYER CULLING

見えないものは描かない。

壁の裏、地下、複雑な建造物の陰にいる動的エンティティを多層空間スキャンで特定し、描画命令の手前で除外します。

03 / COMPACT WORLD DATA

世界を軽く保持する。

頂点・法線・テクスチャ情報を高密度化し、繰り返される形状を共有して VRAM と RAM の使用量を抑える設計です。

従来の最適化は、地形の描画負荷を減らすことには長けていても、モブ・チェスト・看板・アイテムフレームのような動的オブジェクトを、壁や地下の奥にいても描画候補として残してしまう場面があります。RsGraphics は、描画を GPU へ送る一歩前に空間を読むことで、この「見えていないのに処理している」仕事を減らすことを目指します。

高密度なワールドでは、描画速度だけでなく VRAM とシステムメモリの使い方も重要です。頂点・法線・テクスチャ情報をコンパクトに保持し、同じ形状を共有することで、描画距離やリソースパックを使う場面でも余白を残す設計です。Rsift コアのオフヒープ・ブリッジと連携し、メッシュやカリング用データを Java ヒープに載せすぎないことを重視しています。

03 / OFFICIAL PLUGIN

← 概要に戻る

RsCalc.

物理、AI、経路探索、レッドストーン、流体。Minecraft のホットパスをネイティブ計算レイヤーとして再設計する Rsift 公式プラグインです。RsGraphics が「見た目の余裕」を作るのに対し、RsCalc は「 tick の余裕」を作ります。

01 / COLLISION & PHYSICS

連続衝突と高精度積分。

SweptVoxelCollider と 3-Substep Euler/Verlet で、1,024 体規模の衝突・段差昇行・物理更新を扱います。

02 / AI & PATHFINDING

群れ AI と階層型探索。

FlockingAiEngine の全コア並列処理と、Hierarchical A* + JPS の長距離ルート導出を組み合わせます。

03 / WORLD SIMULATION

ワイヤーと流体の計算。

BitsetWireTopology と HydrodynamicEngine で、レッドストーン伝播と水・溶岩のセル計算を扱います。

RsCalc が扱うのは、フレームごと・tick ごとに何度も呼ばれる「小さな計算の積み重ね」です。衝突、物理、AI、経路探索、配線、流体のような処理は、対象が増えるほど Java のスカラー処理とメモリ確保の負担が積み上がります。

そこで空間ハッシュ、階層型グラフ、64-bit ワード演算、全コア並列といった計算構造を用途ごとに使い分けます。目的は特定の数値を競うことだけではなく、重いワールドでも tick 処理の余白を保ち、ゲームの応答を安定させることです。
